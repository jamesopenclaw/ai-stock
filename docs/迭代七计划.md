# 迭代七计划

**文档状态**：执行计划  
**创建日期**：2026-03-29  
**关联迭代**：[多账户支持迭代计划](./多账户支持迭代计划.md)

---

## 1. 目标

在多账户主链路已经打通的前提下，把系统从“能用”推进到“更稳、更容易验收”。

本迭代重点处理三类问题：

- 冷启动或首个请求偏慢
- 关键验收路径缺少可重复的回归约束
- 线上问题出现时，缺少足够直接的定位线索

---

## 2. 范围

- 登录与首个 API 请求稳定性
- 关键页面冷启动体验
- 回归路径固化
- 轻量可观测性补强

不在本迭代范围内：

- 新业务功能扩展
- 新权限模型扩展
- 新数据模型重构

---

## 4. 后续子任务

### 4.1 稳定性

- 继续确认冷启动慢点是否来自容器网络、数据库连接池或首次依赖初始化
- 评估是否需要在后端启动后主动做一次 DB 预热
- 检查首个 `health` 请求偏慢是否与登录慢属于同一根因

### 4.2 可观测性

- 给登录、刷新 token、账户切换等关键链路增加耗时日志
- 为关键页面接口增加更清晰的失败日志标签
- 区分“业务失败”和“冷启动慢”的日志表现

### 4.3 自动化回归

- 补管理员与普通用户两套最小化 UI smoke 流程
- 固化“登录 -> Dashboard -> Review -> Pools/Buy 跳转”的浏览器回归路径
- 把迭代六权限与回归清单转成更接近自动化的执行步骤

## 5. 验收标准

- 冷启动后首次登录不再出现明显卡顿或超时
- 普通用户与管理员的关键权限路径可重复回归
- 至少有一套浏览器级 smoke 流程覆盖登录、权限、账户切换、复盘跳转
- 出现登录慢或接口慢时，日志能快速区分慢点位置

---

## 6. 当前执行建议

建议按以下顺序推进：

1. 先继续定位冷启动慢点
2. 再补关键链路耗时日志
3. 再补管理员切账户和登录流程 smoke
4. 最后把剩余人工回归项整理成执行记录
