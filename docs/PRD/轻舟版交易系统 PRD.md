# 轻舟版交易系统 PRD

**文档版本：V1.0**
**文档类型：产品需求文档（PRD）**
**适用范围：A 股短线交易决策支持**
**主题范围：选股流程、买点分析、卖点分析、账户适配、风险控制、结构化输出**

------

## 1. 产品概述

### 1.1 产品名称

轻舟版交易系统

### 1.2 产品定位

轻舟版交易系统是一套面向 **A 股短线交易场景** 的决策支持系统，围绕"**市场环境判断—板块扫描—个股筛选—买点分析—卖点分析—账户适配—执行输出**"构建完整的交易决策链路。

系统的核心目标不是生成强股名单，而是输出 **结合轻舟账户约束、符合 A 股 T+1 规则、具备实际执行价值的交易计划**。

### 1.3 产品目标

本系统需实现以下目标：

1. 建立每日选股流程
2. 建立盘中/盘前买点判断机制
3. 建立持仓卖点与处理机制
4. 建立基于账户结构的可执行过滤机制
5. 建立统一风控与证伪退出机制
6. 输出结构化、模板化、可验收的交易结果

### 1.4 适用市场与约束

- 市场范围：A 股
- 交易规则：必须遵守 T+1
- 使用对象：轻舟账户
- 输出性质：决策支持，不直接替代人工最终下单判断

## 2. 核心设计原则

### 2.1 账户适配优先

系统在给出任何交易建议前，必须同时回答以下问题：

- 市场上谁最强
- 当前账户能不能参与
- 参与后是否会破坏仓位结构、节奏和纪律

即：**市场强不等于账户可做**。

### 2.2 三池分离

每日选股结果必须固定拆分为三类，不允许混写：

#### 2.2.1 市场最强观察池

仅从市场角度出发，观察最强方向、最强标的及其持续性，不考虑账户约束。

#### 2.2.2 轻舟当前账户可参与池

在市场强度基础上进一步叠加账户约束后，得到的可执行候选池。

#### 2.2.3 持仓处理池

专门用于处理已有持仓，输出持有、减仓、卖出、观察等建议。

### 2.3 市场环境优先

系统必须先判断市场环境，再进行个股层面分析。

市场环境作为全局策略开关，决定：

- 是否允许主动出手
- 是否允许突破型策略
- 买点确认阈值是否上调
- 是否应以防守、腾挪流动性为主

### 2.4 不机械交易

系统不允许使用"价格过线即买入"的机械逻辑。
所有买点必须结合：

- 环境
- 板块
- 个股结构
- 量价关系
- 承接情况
- 账户状态

### 2.5 风险先于机会

弱市先防守，强市再进攻。
持仓风险与流动性风险优先级高于新开机会。

## 3. 用户场景与核心问题

### 3.1 主要使用场景

1. 每日收盘后生成次日交易计划
2. 盘前审视是否存在可执行买点
3. 持仓分析与卖点管理
4. 弱市下的仓位收缩与去弱留强
5. 盘中出现异动后快速校验是否属于可执行机会

### 3.2 系统要解决的核心问题

#### 选股层面

- 市场真正值得看的方向是什么
- 哪些强票只是观察，不应参与
- 哪些机会适合轻舟当前账户执行

#### 买点层面

- 今天能不能买
- 什么时候买
- 什么条件满足才能买
- 买错了如何退出

#### 卖点层面

- 什么时候止损
- 什么时候止盈
- 什么时候减仓
- 什么时候继续观察持有
- 如何区分正常分歧与走弱确认

## 8. 模块需求说明

## 8.1 市场环境模块

### 8.1.1 模块目标

先决定今天是否有必要积极出手，为后续买点与账户建议提供统一环境结论。

### 8.1.2 判定逻辑

#### 倾向防守

满足以下特征时，倾向输出防守：

- 指数普跌
- 情绪差
- 炸板率高
- 跌停增多
- 假突破风险高

#### 倾向进攻

满足以下特征时，倾向输出进攻：

- 指数共振
- 主线清晰
- 市场情绪回暖
- 资金愿意追强
- 强势方向具备持续性

#### 倾向中性

满足以下特征时，倾向输出中性：

- 市场分化明显
- 仍有局部热点
- 尚未全面走坏
- 但不适合无差别进攻

### 8.1.3 输出要求

必须输出：

- market_env_tag
- breakout_allowed
- risk_level
- market_comment

### 8.1.4 研发要求

后续买点分析必须引用该模块输出结果，禁止各模块独立定义环境。

------

## 8.2 板块扫描模块

### 8.2.1 模块目标

识别真正值得关注的方向，而不是只识别当天涨得最多的方向。

### 8.2.2 判定维度

- 当日强度
- 连续活跃性
- 催化因素
- 后续演绎空间
- 是否接近高潮或退潮
- 是否存在证伪信号

### 8.2.3 输出要求

每个板块至少输出：

- 是否主线
- 连续性标签
- 交易性标签
- 证伪线索
- 板块简评

------

## 8.3 个股筛选模块

### 8.3.1 模块目标

从有效板块中筛出可跟踪和可交易的标的。

### 8.3.2 个股初筛条件

至少满足以下条件：

- 属于有效方向，不是无主线漂移股
- 有一定成交支撑
- 有辨识度
- 结构未明显走坏
- 能说明为什么强

### 8.3.3 优先剔除条件

以下情况优先剔除：

- 杂毛脉冲
- 无板块支撑
- 已明显末端加速
- 纯消息一日游
- 账户不适合参与

### 8.3.4 输出要求

每只个股至少输出：

- 强弱标签
- 连续性标签
- 核心属性
- 证伪条件
- 所属池标签

------

## 8.4 买点分析模块

### 8.4.1 模块目标

系统要回答的不是"这只票好不好"，而是：

- 今天能不能买
- 什么时候买
- 什么条件下可以买
- 买错了怎么认错退出

### 8.4.2 买点前置条件

只有同时满足以下条件，才允许进入买点分析：

#### 市场条件

- 市场环境未明显恶化
- 弱市下提高门槛
- 弱市原则上少做普通突破

#### 板块条件

- 板块有强度或回流迹象
- 不是单兵作战
- 至少存在板块或情绪共振

#### 个股条件

- 个股具备辨识度
- 结构未坏
- 非高位末端接盘
- 不是仅因涨得好看而入选

#### 账户条件

- 当前总仓位允许继续出手
- 弱票老仓应优先处理
- 不因可用资金充足而乱开仓

### 8.4.3 买点类型

系统至少支持以下买点类型：

#### 突破买点

必须同时满足：

- 价格突破关键位
- 成交/量能配合
- 有承接
- 板块或指数共振
- 环境允许

#### 回踩承接买点

必须同时满足：

- 趋势未坏
- 回踩位置合理
- 承接清晰
- 回踩后出现再转强迹象

#### 修复转强买点

必须同时满足：

- 前期存在分歧
- 修复不是弱反抽
- 修复后重新获得主动性

#### 低吸买点

当前标记为待确认策略，若纳入必须限制：

- 仅限主线核心
- 仅限明确支撑位
- 仅限环境允许时
- 禁止把接飞刀包装成低吸

### 8.4.4 买点过滤器

所有买点必须经过以下过滤：

- 环境过滤
- 强弱过滤
- 位置过滤
- 量价过滤
- 共振过滤
- 账户过滤

### 8.4.5 买点输出字段

| 字段名           | 类型   | 枚举/说明                                 |
| ---------------- | ------ | ----------------------------------------- |
| buy_signal_tag   | enum   | 可买 / 观察 / 不买                        |
| buy_point_type   | enum   | 突破 / 回踩承接 / 修复转强 / 低吸[需确认] |
| buy_trigger_cond | string | 触发条件                                  |
| buy_confirm_cond | string | 确认条件                                  |
| buy_invalid_cond | string | 失效条件                                  |
| buy_risk_level   | enum   | 高 / 中 / 低                              |
| buy_account_fit  | enum   | 适合 / 一般 / 不适合                      |
| buy_comment      | string | 买点说明                                  |

### 8.4.6 输出规则

每个候选买点建议至少输出：

- 买点类型
- 触发条件
- 参与条件
- 失效条件
- 风险等级
- 是否适合轻舟当前账户

无失效条件的买点，不得进入可执行建议。

## 8.5 卖点分析模块

### 8.5.1 模块目标

卖点模块负责解决：

- 什么时候止损
- 什么时候止盈
- 什么时候减仓
- 什么时候继续持有观察
- 如何识别正常分歧与走弱确认

### 8.5.2 卖点分类

#### 止损卖点

适用于：

- 原始买入逻辑失效
- 跌破关键结构位
- 板块走弱且个股掉队
- 突破失败确认
- 市场环境恶化后个股失去防守价值

要求：

- 失败突破要尽快承认
- 不允许把小亏拖大

#### 止盈卖点

适用于：

- 个股进入加速末端
- 脱离合理博弈区
- 高位异常放量
- 预期兑现、边际收益下降
- 承接不足

#### 减仓卖点

适用于：

- 个股没完全坏，但性价比下降
- 仓位偏重
- 出现更高优先级机会
- 市场转弱，需要回收流动性

#### 观察持有

适用于：

- 趋势未坏
- 回踩有承接
- 板块仍有持续性
- 分歧属于健康分歧

### 8.5.3 持仓处理优先级

系统默认遵循以下顺序：

1. 先处理弱票、低效仓
2. 再考虑是否开新仓
3. 老仓未理顺时，不鼓励多线分散试错

### 8.5.4 卖点判断核心维度

每只持仓票至少检查：

- 是否仍属于主线或强势方向
- 是否仍有板块共振
- 是否跌破关键结构位
- 是否从强转弱
- 是否演变成弱反抽
- 是否受 T+1 约束导致次日处理困难
- 是否影响账户整体节奏
- 原始持有理由是否仍成立

### 8.5.5 持仓票分析模板

建议固定输出：

- 当前状态：强 / 中 / 弱
- 持有理由是否仍成立
- 核心观察位
- 失效信号
- 操作建议：持有 / 减仓 / 卖出 / 继续观察

### 8.5.6 卖点输出字段

| 字段名            | 类型   | 枚举/说明                     |
| ----------------- | ------ | ----------------------------- |
| sell_signal_tag   | enum   | 持有 / 减仓 / 卖出 / 观察     |
| sell_point_type   | enum   | 止损 / 止盈 / 减仓 / 失效退出 |
| sell_trigger_cond | string | 卖点触发条件                  |
| sell_reason       | string | 卖出/减仓原因                 |
| sell_priority     | enum   | 高 / 中 / 低                  |
| sell_comment      | string | 卖点说明                      |

## 8.6 账户适配模块

### 8.6.1 模块目标

把"市场能做"转化为"轻舟账户能不能做"。

### 8.6.2 核心判断问题

系统必须先回答：

- 当前仓位是否允许新仓
- 当前弱票是否应优先处理
- 当前可用资金是否支持继续分仓试错
- 当日是否已经出手过多
- 是否会因为 T+1 导致次日被动

### 8.6.3 输出字段

| 字段名                | 类型    | 说明                       |
| --------------------- | ------- | -------------------------- |
| account_action_tag    | enum    | 可执行 / 谨慎执行 / 不执行 |
| position_pressure_tag | enum    | 低 / 中 / 高               |
| new_position_allowed  | boolean | 是否允许新开仓             |
| priority_action       | string  | 当前优先动作               |
| account_comment       | string  | 账户适配说明               |

### 8.6.4 模块规则

账户适配模块必须作为交易建议落地前的最后一道关口。
即使市场有机会，只要账户不适合，也不能进入可执行建议。

------

## 9. 风控需求

### 9.1 硬约束

系统必须纳入以下硬约束：

- 遵守 A 股 T+1
- 不允许脱离可用资金谈执行
- 不允许在弱市中无差别做突破
- 不允许把普通信号当高确定性信号
- 不允许缺少失效条件
- 不允许无上限分散试错

### 9.2 风控提醒机制

当出现以下情况时，系统应主动提醒：

- 市场转弱后仍频繁开仓
- 连续做失败突破
- 买点仅满足价格过线
- 持仓弱票未处理却继续加新仓
- 出手频率过高
- 非最强核心却给了进攻性仓位

------

## 10. 输出需求

### 10.1 文档输出结构

每次分析建议固定输出以下结构：

#### A. 市场环境

- 今日环境定性
- 是否允许主动出手

#### B. 主线与板块扫描

- 主线候选
- 次主线候选
- 证伪点

#### C. 三池输出

1. 市场最强观察池
2. 轻舟当前账户可参与池
3. 持仓处理池

#### D. 买点分析

- 每只候选票的买点类型
- 触发条件
- 失效条件
- 参与前提

#### E. 卖点分析

- 每只持仓票的持有逻辑
- 失效信号
- 操作建议

#### F. 执行摘要

用最短的话说明：

- 今天该不该出手
- 优先看谁
- 先处理谁
- 哪些绝对不碰

## 11. 验收标准

### 11.1 选股模块验收

- 能否区分市场最强观察池和账户可参与池
- 能否识别持仓处理池
- 能否输出证伪条件
- 能否避免把老熟票默认塞回结果

### 11.2 买点模块验收

- 是否仍在用"过价即买"的偷懒逻辑
- 是否引用市场环境结果
- 是否区分突破、回踩、修复
- 是否提供失效条件

### 11.3 卖点模块验收

- 是否能区分止损、止盈、减仓、观察
- 是否能识别失败突破
- 是否优先处理弱票、低效仓
- 是否考虑 T+1

### 11.4 账户模块验收

- 是否真正考虑可用资金、总仓位、已有持仓
- 是否能限制弱市乱开仓
- 是否能给出"先处理谁"的优先级

------

## 12. 当前待确认项

以下内容需单独确认，避免开发阶段自行补脑：

1. 连续性评分、交易性评分是否需要数值化输出
2. 市场环境标签固定为 3 档还是扩展为 5~7 档
3. 低吸策略是否正式纳入
4. 仓位建议是否由系统直接给出百分比
5. 止损/止盈是否允许输出具体数值
6. 盘中买点是否需要分钟级数据支撑
7. 是否需要显式区分"按计划失败突破"与"情绪化追高"
8. 持仓处理池是否加入"优先卖出顺序"字段
9. 是否需要形成模板化验收字段

------

## 13. 开发高风险误区

开发过程中最容易出现的问题包括：

1. 把强股榜当成选股系统
2. 把价格过线当成买点确认
3. 只写"建议关注"，不写触发和失效
4. 不看账户结构，只看市场热度
5. 不区分观察池、可执行池、持仓池
6. 卖点只会写"逢高止盈"，缺少结构判断
7. 弱市里默认突破优先，导致系统性回撤

------

## 14. 一句话验收标准

如果系统最终输出的仍然只是：

- 一堆强股名单
- 没有账户适配
- 没有买点触发条件
- 没有卖点失效条件
- 没有市场环境开关
- 没有持仓优先级

那么它不应被定义为"轻舟版交易系统"，而只是一个会复述涨幅榜的提示器。
