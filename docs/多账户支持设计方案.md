# 多账户支持设计方案

**文档状态**：基线方案  
**创建日期**：2026-03-29  
**适用范围**：轻舟版交易系统多账户改造  
**用途**：作为后续设计、开发、测试、验收的统一依据，避免实现偏离目标

---

## 1. 背景

当前系统本质上是单账户实现：

- 持仓数据没有账户维度，默认全局读取
- 账户配置是单例配置
- 决策分析直接依赖全局持仓和全局账户状态
- 三池快照和复盘快照中混入了账户相关结果
- Web 端与小程序端都没有登录体系、角色体系、权限控制

本次改造目标不是单纯增加登录页，而是将系统从“单账户工具”演进为“支持多用户、多交易账户、账户数据隔离、市场数据共享”的系统。

---

## 2. 目标

### 2.1 业务目标

1. 增加登录功能，支持管理员和普通用户
2. 管理员可以管理用户和交易账户
3. 每个交易账户的数据严格隔离
4. 市场环境和板块扫描数据在所有账户之间共享
5. 在共享市场分析的前提下，账户可参与池、持仓处理、账户状态等结果按账户独立计算

### 2.2 设计目标

1. 明确“用户”和“交易账户”是两个不同概念
2. 明确“共享数据”和“隔离数据”的边界
3. 保证后续接口、缓存、快照、日志都不会发生跨账户串数据
4. 控制改造复杂度，按阶段推进，支持平滑迁移

---

## 3. 核心原则

### 3.1 用户与交易账户分离

- `用户(User)` 用于登录、鉴权、权限控制
- `交易账户(TradingAccount)` 用于持仓、资产、账户状态、决策分析
- 一个普通用户默认绑定一个交易账户
- 管理员可以管理多个交易账户，并支持切换查看

### 3.2 共享市场上下文，隔离账户上下文

共享：

- 市场环境
- 板块扫描
- 行情原始数据
- 候选股票基础池
- 市场观察池

隔离：

- 持仓
- 账户配置
- 账户概况
- 账户状态
- 账户可参与池
- 持仓处理池
- 买点分析
- 卖点分析
- 复盘快照
- LLM 缓存与调用日志中带账户语境的部分

### 3.3 后续变更必须以本文为基线

- 涉及多账户功能时，优先检查本文定义的数据边界和职责分层
- 若实现方案需要偏离本文，必须先更新本文，再进行代码改造
- 任何“临时兼容写法”不得破坏账户隔离原则

---

## 4. 领域模型

### 4.1 用户

用于登录和权限控制。

建议字段：

- `id`
- `username`
- `password_hash`
- `display_name`
- `role`：`admin | user`
- `status`：`active | disabled`
- `last_login_at`
- `created_at`
- `updated_at`

### 4.2 交易账户

用于表达业务上的交易实体。

建议字段：

- `id`
- `account_code`
- `account_name`
- `owner_user_id`
- `status`
- `created_at`
- `updated_at`

### 4.3 账户设置

用于保存每个交易账户自己的配置。

建议字段：

- `account_id`
- `total_asset`
- `risk_profile`
- `llm_enabled`
- `llm_provider`
- `llm_base_url`
- `llm_api_key`
- `llm_model`
- `llm_timeout_seconds`
- `llm_temperature`
- `llm_max_input_items`
- `updated_at`

### 4.4 持仓

现有 `holdings` 必须增加账户维度。

关键要求：

- 增加 `account_id`
- 所有持仓查询必须带 `account_id`
- 所有持仓增删改必须以 `account_id` 为前提

---

## 5. 数据边界

### 5.1 共享数据

以下数据在所有账户之间共享，不按账户拆分：

- 市场环境分析结果
- 板块扫描结果
- 原始行情数据
- 候选股票原始列表
- 股票基础评分结果
- 市场观察池

### 5.2 账户隔离数据

以下数据必须严格按账户隔离：

- 持仓明细
- 总资产与账户参数
- 账户概况与账户状态
- 账户可参与池
- 持仓处理池
- 买点分析结果
- 卖点分析结果
- 复盘快照
- LLM 缓存
- LLM 调用日志

### 5.3 明确禁止

以下做法禁止继续存在：

- 用全局单例账户配置代表所有账户
- 用全局持仓代表当前登录用户的持仓
- 将 `account_executable_pool` 缓存为全局快照
- 将含账户语境的复盘快照按交易日全局覆盖

---

## 6. 系统分层

### 6.1 共享层

职责：

- 计算市场环境
- 计算板块扫描
- 获取候选股票
- 计算股票基础评分
- 输出市场观察池

特点：

- 与用户、账户无关
- 可以全局缓存
- 可以按交易日复用

### 6.2 账户层

职责：

- 加载当前账户持仓
- 加载当前账户配置
- 构建账户画像
- 生成账户可参与池
- 生成持仓处理池
- 输出买点分析、卖点分析、账户适配结果

特点：

- 强依赖 `account_id`
- 不可使用全局结果替代
- 可复用共享层输入，但结果必须按账户单独缓存

---

## 7. 数据库设计方向

建议新增表：

- `users`
- `trading_accounts`
- `account_settings`
- `refresh_tokens` 或 `user_sessions`

建议改造表：

- `holdings`：新增 `account_id`
- `review_snapshots`：新增 `account_id`
- `llm_cache_entries`：新增 `account_id` 或 `scope_type/scope_id`
- `llm_call_logs`：新增 `account_id`
- `stock_pool_snapshots`：不得继续只按 `trade_date + limit` 全局存储完整三池结果

建议保留为共享表：

- `sector_scan_snapshots`
- 市场环境相关快照
- 原始行情缓存

---

## 8. 接口设计原则

### 8.1 认证接口

新增：

- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`
- `POST /api/v1/auth/logout`
- `GET /api/v1/auth/me`

### 8.2 管理员接口

新增：

- 用户管理
- 交易账户管理
- 账户绑定关系管理
- 重置密码、禁用账户、启用账户

### 8.3 业务接口

现有以下接口都必须改造为“基于当前账户上下文”：

- `/api/v1/account/*`
- `/api/v1/decision/*`
- `/api/v1/stock/pools`

普通用户：

- 默认只能访问自己绑定账户的数据

管理员：

- 可以查看和管理所有账户
- 如支持切换查看账户，必须显式指定目标 `account_id`

---

## 9. 权限模型

### 9.1 管理员

权限：

- 登录后台
- 查看全部用户
- 创建、编辑、禁用用户
- 创建、编辑、禁用交易账户
- 绑定用户和交易账户
- 查看任意账户的数据
- 维护系统级配置

### 9.2 普通用户

权限：

- 登录系统
- 查看自己的账户
- 管理自己的持仓
- 查看自己的决策分析结果
- 不可访问用户管理和账户管理能力
- 不可查看其他账户数据

---

## 10. 前端改造原则

### 10.1 Web 端

必须增加：

- 登录页
- 登录态存储
- 路由守卫
- 401 统一处理
- 当前用户信息
- 当前账户上下文

管理员增加：

- 用户管理页
- 交易账户管理页
- 账户切换查看能力

普通用户：

- 保留现有业务页
- 默认只访问自己的账户

### 10.2 小程序端

必须增加：

- 登录页
- token 持久化
- 请求头鉴权
- 401 失效处理
- 当前账户上下文

---

## 11. 缓存与快照原则

### 11.1 可以共享缓存的结果

- 市场环境快照
- 板块扫描快照
- 候选股票池快照
- 市场观察池

### 11.2 必须账户隔离的缓存

- 账户可参与池
- 持仓处理池
- 买点结果
- 卖点结果
- 复盘快照
- LLM 缓存
- LLM 日志

### 11.3 特别说明

当前完整三池结果中同时包含：

- `market_watch_pool`
- `trend_recognition_pool`
- `account_executable_pool`
- `holding_process_pool`

其中前两者偏共享，后两者偏账户隔离。因此后续不得再将完整三池结果整体按全局缓存，而应拆分为：

- 共享部分
- 账户覆盖部分

---

## 12. 迁移策略

采用分阶段迁移，不建议长期维持单账户和多账户双逻辑并存。

### 12.1 迁移步骤

1. 新增用户、交易账户、账户设置、会话相关表
2. 为 `holdings` 增加 `account_id`
3. 为快照和日志表增加账户维度
4. 创建默认管理员
5. 创建默认普通用户
6. 创建默认交易账户
7. 将现有全量持仓迁移到默认交易账户
8. 将现有单例账户配置迁移为默认账户设置
9. 完成代码切换后，移除单例账户依赖

### 12.2 迁移要求

- 迁移脚本必须可重复执行或具备幂等保护
- 迁移完成后，系统内不允许继续存在“无账户归属”的持仓数据
- 所有旧代码中默认读取全局持仓或单例账户配置的逻辑必须清理

---

## 13. 分阶段实施计划

### P1：认证与基础表

- 新增用户体系
- 新增登录接口
- 新增角色权限依赖
- 完成数据库迁移基础设施

### P2：账户隔离

- `holdings` 按账户读写
- 账户配置按账户读取
- 账户相关 API 全部切换到 `account_id`

### P3：决策上下文拆层

- 拆分共享上下文与账户上下文
- 共享市场分析结果复用
- 账户结果单独计算

### P4：缓存与快照治理

- 三池快照拆分
- 复盘快照拆分
- LLM 缓存和日志增加账户维度

### P5：管理后台

- 用户管理
- 交易账户管理
- 账号禁用与重置密码

### P6：双端联调

- Web 登录与路由权限
- 小程序登录与鉴权
- 管理员与普通用户流程联调

### P7：测试与验收

- 单元测试
- API 集成测试
- 多账户隔离测试
- 管理员权限测试

---

## 14. 验收标准

### 14.1 功能验收

- 管理员可以创建和管理用户
- 管理员可以创建和管理交易账户
- 普通用户可以登录并访问自己的账户
- 普通用户不能访问其他账户数据
- 管理员可以查看指定账户数据

### 14.2 数据隔离验收

- 不同账户的持仓互不可见
- 不同账户的账户状态互不可见
- 不同账户的 `account_executable_pool` 不会串数据
- 不同账户的复盘快照不会互相覆盖
- 不同账户的 LLM 缓存和日志不会混用

### 14.3 共享数据验收

- 市场环境在不同账户下结果一致
- 板块扫描在不同账户下结果一致
- 候选股基础结果在不同账户下结果一致

---

## 15. 后续执行约束

后续所有与多账户相关的开发，均需满足以下约束：

1. 先判断该数据是共享还是隔离
2. 若与持仓、资金、账户状态有关，则默认按账户隔离
3. 若新增缓存、快照、日志表，必须明确是否需要 `account_id`
4. 若新增接口，必须明确鉴权方式、角色权限、账户归属
5. 若方案与本文冲突，应先修改本文，再修改实现

---

## 16. 当前结论

本项目多账户改造的标准路径是：

- 先建立用户与交易账户体系
- 再实现账户隔离
- 再拆分共享分析层与账户分析层
- 最后治理缓存、快照、权限和前端交互

后续实现不得退回到“全局单例账户 + 登录壳子”的伪多账户方案。
