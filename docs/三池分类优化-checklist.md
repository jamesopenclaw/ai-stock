# 三池分类优化 Checklist

## Epic
**优化选股逻辑（三池分类）：提升盘中适配度，避免“过于静态、过于收口”**

---

# 一、目标
本轮优化的核心目标不是让系统更激进，而是解决这几个问题：

- **主线优先，不等于只能盯唯一主线**
- **代表票优先，不等于压掉所有真实可交易机会**
- **账户保护优先，不等于丧失进攻试错能力**
- **观察池应该像雷达，不只是像排行榜**
- **账户池应该像执行清单，不只是像最稳妥清单**

---

# P0｜先修结构性偏保守问题

---

## 1. 放宽“唯一主线入池”机制
**文件：** `app/services/stock_filter.py`

### 当前问题
新开仓三池逻辑过度收口：
- 有 `A_MAINLINE` 就只允许 A
- 没有 A 才允许 B
- 方向切换、新主线萌芽期容易被错过

### 任务
- [ ] 检查 `_resolve_new_pool_profiles()`
- [ ] 调整逻辑：主线优先，但允许保留 **1 个新强化方向**
- [ ] 增加“候补强化方向”概念，不必直接进账户池，但可以进入观察池
- [ ] 不再让旧主线天然垄断全部新开仓池子

### 建议方向
- [ ] 保留当前最强 A 主线
- [ ] 额外允许 1 个“强度快速提升/盘中强化”的次方向进入 `market_watch` 或 `trend_recognition`
- [ ] 该方向不必立即放入 `account_executable`

### 验收
- [ ] 当市场出现方向切换时，系统能保留一个新方向观察席位
- [ ] 不再出现“旧主线还在榜上，所以新方向完全看不到”的情况

---

## 2. 代表票机制按池子分层，而不是一刀切
**文件：** `app/services/stock_filter.py`

### 当前问题
`_select_representative_codes()` + `_pick_sector_representatives()` 在前面就把板块样本压成 2~4 只，导致：
- 市场池变干净，但执行池样本也被压扁
- “有辨识度的票”和“最好买的票”被混为一谈

### 任务
- [ ] 拆分代表票机制在不同池子的使用方式
- [ ] `market_watch_pool`：允许强代表化
- [ ] `trend_recognition_pool`：允许半代表化
- [ ] `account_executable_pool`：不能完全受代表票约束
- [ ] 保留“非代表但买点更舒服”的票进入账户池的机会

### 建议方向
- [ ] 为 `account_executable_pool` 新增一条独立候选支路
- [ ] 允许同板块中的“执行优先票”绕过代表票截断
- [ ] 将“辨识度排序”和“执行舒适度排序”解耦

### 验收
- [ ] 市场观察池依然简洁
- [ ] 账户池不再只剩“最像龙头的票”，而能出现“最适合出手的票”

---

## 3. 增加“盘中强化候选槽位”
**文件：** `app/services/stock_filter.py`  
**可能涉及：** `decision_context.py` / 数据输入层

### 当前问题
观察池主要基于静态评分和已有板块地位，盘中刚强化的方向、刚转强的票不容易被纳入。

### 任务
- [ ] 为 `market_watch_pool` 或 `trend_recognition_pool` 增加少量“盘中强化候选槽位”
- [ ] 允许 1~2 只不完全符合静态主线标签的票进入观察
- [ ] 增加“盘中强化”判定字段或占位逻辑

### 建议可用特征
- [ ] 午后重新站上均价线
- [ ] 日内回落后重新走强
- [ ] 板块联动人数增加
- [ ] 量能较开盘阶段明显放大
- [ ] 盘中从非前排切入前排

### 验收
- [ ] 观察池不再只像复盘排行榜
- [ ] 能捕捉到“当天刚强化”的潜在机会

---

## 4. 避免账户池过度“只要舒服，不要强度”
**文件：** `app/services/stock_filter.py`

### 当前问题
账户池偏好：
- 回踩确认
- 低吸预备
- 舒服位置
- 不追突破

这有助于纪律，但会压制真正有进攻价值的机会。

### 任务
- [ ] 检查 `_should_enter_account_pool()`
- [ ] 检查 `_has_comfortable_account_entry()`
- [ ] 检查 `_passes_account_participation_gate()`
- [ ] 放宽“可参与池只能收舒服票”的倾向
- [ ] 给“强度足够但位置不完美”的票保留小仓试错资格

### 建议方向
- [ ] 保留原有“标准执行票”逻辑
- [ ] 增加“进攻试错票”逻辑分支
- [ ] 对真正强方向中的突破票，不要因为“稍微不舒服”就一票否决

### 验收
- [ ] 账户池既有舒服买点的票，也允许少量强势试错票
- [ ] 不再出现“所有强票都因为不够舒服被排除”的情况

# P1｜优化池子边界和分类语义

---

## 5. 重构 market_watch_pool 的定位：从“标准答案池”升级为“机会雷达池”
**文件：** `app/services/stock_filter.py`

### 当前问题
`market_watch_pool` 现在更像“今天看起来最标准的强票集合”，不够像“值得持续关注的热点雷达”。

### 任务
- [ ] 检查 `_should_enter_market_watch()`
- [ ] 检查 `_market_watch_vote_count()`
- [ ] 放宽“观察池”对静态完美形态的依赖
- [ ] 允许部分“未完全成熟但明显强化”的票进入

### 建议方向
- [ ] market_watch 接纳两类票：
  - [ ] 已被证明的最强票
  - [ ] 正在快速强化、值得盯的票
- [ ] 不要只让“标准强票”进观察池

### 验收
- [ ] 观察池兼具“最强票”与“潜在强化票”
- [ ] 用户从观察池能看到热点雷达，而不是只看到定稿答案

---

## 6. 重构 trend_recognition_pool 的定位：从“耐看票池”升级为“有延续观察价值的票池”
**文件：** `app/services/stock_filter.py`

### 当前问题
趋势辨识度池当前逻辑偏合理，但和 `market_watch_pool` 仍有一定重叠，缺少“为什么它值得继续看”的独立标准。

### 任务
- [ ] 检查 `_should_enter_trend_recognition()`
- [ ] 检查 `_trend_recognition_vote_count()`
- [ ] 区分“今天最强”和“未来两天可能继续有辨识度”
- [ ] 提升“趋势辨识度池”的独立存在感

### 建议方向
- [ ] 更强调：
  - [ ] 结构完整性
  - [ ] 连续性
  - [ ] 次日可交易性
  - [ ] 是否具备板块映射价值
- [ ] 弱化纯日内涨幅对 trend 池的影响

### 验收
- [ ] trend 池和 market 池角色边界更清晰
- [ ] trend 池更像“值得继续盯几天”的票，而不是“第二个观察池”

---

## 7. 重新梳理账户池的“入池原因”与“未入池原因”
**文件：** `app/services/stock_filter.py`  
**可能涉及：** 前端展示字段

### 当前问题
现在虽然已有：
- `why_this_pool`
- `not_other_pools`

但对用户来说，“不进账户池”很容易被误解成“这票不行”。

### 任务
- [ ] 为“未进账户池”增加更细的原因标签
- [ ] 区分以下几类：
  - [ ] 方向强，但买点不舒服
  - [ ] 方向可看，但账户不适配
  - [ ] 方向一般，不值得执行
  - [ ] 可观察，但不值得当下出手
- [ ] 考虑新增类似字段：
  - [ ] `miss_risk_note`
  - [ ] `why_not_executable_but_should_watch`

### 验收
- [ ] 用户能看懂“不进账户池”的真实原因
- [ ] 系统不再默认传达“不入执行池 = 这票没价值”

# P2｜升级底层评分逻辑

---

## 8. 降低静态日线评分对盘中机会的压制
**文件：** `app/services/stock_filter.py`

### 当前问题
当前 `_score_stock()` 和后续入池逻辑，主要依据：
- 涨跌幅
- 换手率
- 量比
- 成交额
- 收盘位置
- 板块主线标签
- 候选来源

这些更像盘后/阶段性总结，不够像盘中决策特征。

### 任务
- [ ] 识别哪些分数属于“复盘强度”
- [ ] 识别哪些分数属于“盘中可交易性”
- [ ] 将二者拆分，避免一个静态总分支配所有池子

### 建议方向
新增两套分：
- [ ] `market_strength_score`：偏市场强度
- [ ] `execution_opportunity_score`：偏交易机会

### 验收
- [ ] 市场观察池更多看强度
- [ ] 账户池更多看执行性
- [ ] 不再所有池都围绕同一总分旋转

---

## 9. 引入“方向切换 / 新强化”识别逻辑
**文件：** `app/services/stock_filter.py` / `app/services/sector_scan.py`

### 当前问题
板块只分：
- 主线
- 次主线
- 跟风
- 杂毛

但缺少对：
- 主线衰减
- 次主线强化
- 新方向萌芽
的专门识别。

### 任务
- [ ] 在板块层增加“强化中 / 切换中 / 衰减中”类状态
- [ ] 在个股层增加“受益于方向切换”的观察逻辑
- [ ] 给池子分类提供更动态的方向信号

### 验收
- [ ] 系统能对“新方向冒头”更敏感
- [ ] 不再死盯静态主线标签

---

## 10. 升级代表角色识别，减少“静态角色误判”
**文件：** `app/services/stock_filter.py`

### 当前问题
角色识别较依赖：
- 涨停
- 涨幅
- 成交额
- 是否核心

但真实交易中的“龙头 / 中军 / 趋势核心”是动态博弈结果，不应只由静态日内指标决定。

### 任务
- [ ] 检查 `_determine_stock_role_tag()`
- [ ] 检查 `_representative_role_tag()`
- [ ] 放宽“角色定义 = 固定标签”的思路
- [ ] 减少角色识别直接主导池子命运的权重

### 建议方向
- [ ] 角色只作为解释字段
- [ ] 不应成为代表票筛选的唯一核心入口
- [ ] 保留“非典型代表票但交易性更好”的进入机会

### 验收
- [ ] 不再因为“不是典型龙头/中军”就失去观察/执行资格
- [ ] 角色识别更多服务解释，不强行绑死交易选择

# 回归测试 Checklist

---

## Case A｜旧主线未死，新方向刚起
### 场景
- 旧主线仍有高分和静态标签优势
- 新方向盘中刚强化，但尚未完全坐稳主线

### 期望
- [ ] 旧主线仍可保留
- [ ] 新方向至少能进入观察池一个席位
- [ ] 不再被旧主线完全压死

---

## Case B｜板块内最有辨识度的票，不是最好买的票
### 场景
- 龙头高高在上
- 第二梯队/换手票更适合账户执行

### 期望
- [ ] 市场池仍能看到龙头
- [ ] 账户池可出现更舒服的执行票
- [ ] 不要求账户池必须与代表票完全一致

---

## Case C｜强票不够舒服，但值得试错
### 场景
- 方向强
- 个股强
- 买点不算理想
- 但具备小仓试错价值

### 期望
- [ ] 可进入账户池的“试错分支”
- [ ] 不再因为“不够舒服”直接失去全部资格

---

## Case D｜观察池不应只剩标准强票
### 场景
- 一些票已被证明最强
- 一些票是盘中刚强化，尚未完全成熟

### 期望
- [ ] 观察池同时看到“确定强票”和“潜在强化票”
- [ ] 观察池更像雷达，不只是榜单

# 一句项目备注
这轮优化不是让系统学会“追涨”，  
而是让它先学会：

## **“主线优先” 不等于 “只有主线能看”；  
## “代表票优先” 不等于 “只有代表票能做”。**
